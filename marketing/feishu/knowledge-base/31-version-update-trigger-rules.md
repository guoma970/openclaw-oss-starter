# 版本更新触发规则

## 目标

明确什么变化会触发版本更新，以及触发后哪些链路需要同步。

## 适用对象

- skill
- 项目首页
- release note
- 知识库项目页
- 对外介绍文案

## 触发类型

### 1. Skill 结构变化

例如：

- 新增 `SKILL.md`
- 新增 `README.md`
- 新增 `agents/openai.yaml`
- 新增 examples
- 新增 customization / release notes

建议：

- 视为版本更新候选
- 至少同步 GitHub
- 若 skill 已公开，则评估是否同步 ClawHub

### 2. Skill 能力变化

例如：

- 新增功能
- 修改工作流
- 修改边界
- 增加新示例

建议：

- 触发 skill 版本更新
- 同步 GitHub
- 若已发布到 ClawHub，则同步 ClawHub
- 如影响理解方式，同步飞书知识库

### 3. 项目定位变化

例如：

- 首页定位变化
- 主推 skill 变化
- 仓库简介变化

建议：

- 同步 GitHub 首页
- 同步飞书知识库总览页与对外介绍页
- 评估是否要刷新对外文案

### 4. 知识库规则变化

例如：

- 新增执行规则
- 新增回报模板
- 新增审批流

建议：

- 至少同步飞书知识库
- 若规则也影响仓库说明，则同步 GitHub

### 5. 对外文案变化

例如：

- 新增平台文案
- 修改项目简介
- 调整对外表达

建议：

- 同步飞书知识库文案页
- 评估是否要同步 GitHub 简介

## 不一定需要版本更新的情况

以下内容通常不需要 skill 版本 bump：

- 纯知识库复盘
- 纯内部流程补充
- 与 skill 本身无关的局部记录

## 最小判断规则

如果变化影响：

- skill 内容
- skill 使用方式
- skill 公开展示

则优先考虑版本更新。

如果变化只影响：

- 内部复盘
- 知识库管理
- 内部协作说明

则优先视为知识库更新，不必强行 bump skill 版本。

## 推荐触发后动作

### Skill 版本更新

```text
更新本地内容
-> 更新 GitHub
-> 视情况更新 ClawHub
-> 视情况更新飞书知识库
-> 视情况刷新对外文案
```

### 规则或说明更新

```text
更新本地内容
-> 更新飞书知识库
-> 如影响对外展示，再补 GitHub
```

## 一句话规则

影响 skill 本身，就考虑版本更新；只影响规则与沉淀，就优先走知识库更新。
