# 公开链路执行顺序

## 目标

把公开链路的执行顺序固定下来，避免 GitHub、ClawHub、飞书知识库、对外文案之间出现先后混乱。

## 适用范围

- 新 skill 发布
- 现有 skill 版本更新
- 项目首页定位调整
- 对外简介更新
- 新增知识库项目页

## 标准顺序

### 1. 本地准备

先由 `main` 完成：

- 本地内容修改
- 结构检查
- public-safe 检查
- 发布前检查
- 仓库校验

只有本地准备完成后，才进入公开链路。

### 2. GitHub

先同步 GitHub，因为它是公开项目主阵地。

典型动作：

- 更新 README
- 更新 skill 文件
- 更新 changelog / release notes
- 提交并 push

## 3. ClawHub

在 GitHub 内容稳定后，再处理 ClawHub。

典型动作：

- 首次发布 skill
- bump skill 版本
- 更新 skill 描述
- 检查可见状态

## 4. 飞书知识库

在 GitHub / ClawHub 主信息稳定后，再同步飞书知识库。

典型动作：

- 更新项目总览
- 更新对外发布文案
- 更新申请与介绍素材
- 更新规则、复盘、回执页

## 5. 对外文案与群内同步

最后处理营销和协同侧同步。

典型动作：

- Feishu 群内同步
- X 短帖
- 小红书改写稿
- 公众号 / 长文引子

## 推荐顺序总结

```text
本地准备
-> GitHub
-> ClawHub
-> 飞书知识库
-> 对外文案 / 群内同步
```

## 特殊情况

### 只改知识库内容

如果本次仅是：

- 复盘
- 账号矩阵
- 执行规则
- 通用指令沉淀

则可以直接：

```text
本地准备
-> 飞书知识库
```

### 只改 GitHub

如果本次只是：

- 本地代码更新
- 不影响公开说明的小改动

则可以只停留在：

```text
本地准备
-> GitHub
```

### skill 不适合公开

如果发布前检查未通过：

- 不进入 GitHub / ClawHub / 飞书公开链路
- 只保留本地或仅进入知识库内部沉淀

## 一句话规则

先本地准备，再先 GitHub、后 ClawHub、再飞书知识库，最后才做对外传播。
