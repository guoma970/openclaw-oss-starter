# 运营群规则公开性评估

## 目标

对当前这组“运营群协同规则”文件做统一评估，判断它们适合：

- 公开发布
- 仅知识库沉淀
- 仅本地保留
- 先抽象再判断

## 当前评估对象

- `config/openclaw-feishu-group-routing.yaml`
- `30_运营群统一协同规则（正式版）`
- `31_运营群适用说明表`
- `32_群内标准消息模板`

## 评估结论

### 1. `config/openclaw-feishu-group-routing.yaml`

结论：

- `C 级：仅本地保留`

理由：

- 含有真实 `chat_id`
- 含有具体群名
- 含有明确 agent / group 路由关系
- 属于系统配置而不是通用公开模板

处理建议：

- 仅本地保留
- 如需公开，必须先彻底改造成不含真实群名和真实 chat_id 的示例模板

### 2. `30_运营群统一协同规则（正式版）`

结论：

- `D 级：先抽象再判断`

理由：

- 有较强的方法论价值
- 但当前仍然带有明显的个人体系命名与场景绑定
- 更适合作为内部规则沉淀或抽象改写基础

处理建议：

- 可以先作为飞书知识库内部方法论沉淀
- 如要公开，需改成通用化版本，去掉特定体系命名

### 3. `31_运营群适用说明表`

结论：

- `C 级：仅本地保留`

理由：

- 包含具体群名
- 群名直接对应真实运营对象
- 不适合作为公开内容直接展示

处理建议：

- 仅本地保留
- 如果要公开，只能改写成通用账号分组示例表

### 4. `32_群内标准消息模板`

结论：

- `B / D 之间，建议按 D 处理`

理由：

- 模板本身有通用价值
- 但当前仍然与前述运营体系绑定
- 若稍作抽象，潜力很大

处理建议：

- 先抽象成通用运营消息模板
- 去掉具体群体系绑定
- 抽象后可进入公开候选池

## 总体建议

当前这组文件里：

- 最不适合公开的是路由配置与适用矩阵
- 最值得抽象后公开的是标准消息模板
- 统一协同规则适合作为内部方法论沉淀，再决定是否公开

## 一句话结论

这组文件目前整体更偏“内部协同规则”，不是直接公开发布包；其中只有消息模板最值得优先抽象成公开版。
