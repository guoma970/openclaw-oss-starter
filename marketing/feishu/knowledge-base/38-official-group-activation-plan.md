# 正式群启用方案

## 目标

让正式协作群从“已经建好”进入“可稳定使用”的状态。

## 群定位

本群定位为：

- OpenClaw 正式运营发布协作群

本群用途：

- GitHub 项目任务入口
- ClawHub 同步协作
- 飞书知识库同步协作
- 对外发布与营销协作
- 发布复盘与执行回报

## 默认上下文

本群默认继承当前项目上下文：

- 默认项目：`openclaw-oss-starter`
- 默认 GitHub 仓库：`guoma970/openclaw-oss-starter`
- 默认工作区：`/Users/ztl/Documents/New project`
- 默认协作链：`main -> 小果 -> 果爸`

因此在本群里，日常短口令不应再频繁要求补 repo 或路径。

## 成员角色

### 你

- 决策
- 提任务
- 定方向

### `main`

- 任务入口
- 任务分类
- 路由分发
- 最终回报

### `小果`

- 知识库同步
- 文案整理
- 公开同步执行

### `果爸`

- 国内平台发布执行
- 尤其是小红书等账号侧动作

## 启用顺序

### 第 1 步：发布群定位说明

先在群里明确：

- 本群不是测试群
- 本群用于正式协作
- 后续 GitHub / ClawHub / 飞书知识库相关任务默认从本群进入

### 第 2 步：做一次试运行

建议第一条正式任务是：

- 日常同步检查

目的：

- 检查 `main -> 小果 -> 果爸` 的协作链是否顺

### 第 3 步：做一次正式发布型任务

可以选择：

- GitHub 项目任务
- 知识库同步
- 新 skill 上线流程
- 多项目日常检查

### 第 4 步：固定日常短指令

让群里以后优先使用：

- 日常同步检查
- 知识库同步
- GitHub 项目处理
- 新 skill 上线流程
- 周度复盘
- 极简短口令

## 推荐首批消息

### 消息 1：群定位

```text
请从 main 建立本群的正式协作规则。
要求：
1. 本群作为 OpenClaw 正式运营发布协作群。
2. main 负责总入口、判断、分流、回报。
3. 小果负责知识库同步、公开同步、文案整理。
4. 果爸负责国内平台实际发布执行。
5. 后续 GitHub、ClawHub、飞书知识库相关任务默认从本群进入。
```

### 消息 2：试运行

```text
请从 main 做一次本群试运行。
要求：
1. 按正式协作群规则执行。
2. 返回本群定位、角色分工、标准任务入口格式。
3. 如需补充规则，直接提出。
```

### 消息 3：正式任务

```text
请从 main 做一次日常同步检查。
```

### 消息 4：正式承接

```text
请从 main 正式承接这条项目工作线。
```

## 启用成功的标志

- `main` 能稳定接任务
- `小果` 能稳定承接同步
- `果爸` 明确知道自己只承接国内平台发布
- 群里的消息开始按短指令和标准回报格式运行
- `main` 已能按项目登记表执行多项目检查
- `main` 已能按脱敏规则回报而不泄露本地结构

## 一句话规则

先定群定位，再试运行，再接正式任务，再固定短指令。
