# 48_Multi-project Daily Check 规则

## 目标

让 `main` 的每日日常检查不再只盯一个公开项目，而是面向当前活跃项目列表。

## 核心原则

`main` 的日常检查对象应是：

- 当前活跃项目
- 每个项目各自的公开边界
- 每个项目各自应同步到哪里

而不是只盯 `openclaw-oss-starter`。

## 当前建议纳入日常检查的项目

### 1. `openclaw-oss-starter`

定位：

- 公开项目
- skill collection
- GitHub / ClawHub / 飞书知识库 都可能需要同步

### 2. `言町驾驶舱`

定位：

- 正在开发中的项目
- 当前不默认进入公开 skill 发布链
- 先以项目推进、规则沉淀、内部协同为主

### 3. `羲果陪伴`

定位：

- 商业化敏感项目
- 不默认进入公开发布链
- 优先按商业化保护原则处理

## main 每日应做什么

`main` 每天应按项目逐个判断：

1. 今天是否有实际工作推进
2. 这些变化属于：
   - 公开 GitHub 更新
   - 飞书知识库沉淀
   - 仅本地保留
   - 商业化敏感，不公开
3. 是否需要派给 `小果`
4. 是否需要等你拍板

## 推荐回报方式

回报时按项目分段，例如：

```text
openclaw-oss-starter:
- 同步范围判断:
- 动作:

言町驾驶舱:
- 同步范围判断:
- 动作:

羲果陪伴:
- 同步范围判断:
- 动作:
```

## 一句话规则

`main daily check` 应是多项目巡检，不是单仓库巡检。
