# 本地工具与私有脚本评估

## 目标

对当前工作区里的本地工具脚本做一次发布性判断，避免误把私有工具直接公开。

## 当前评估对象

- `scripts/vertex/README.md`
- `scripts/vertex/nanobanana.py`
- `scripts/vertex/veo31.py`
- `scripts/vertex/oc-image`
- `scripts/vertex/oc-veo`

## 当前结论

### 总体判断

- 当前建议：`D / C 之间，优先按 C 处理`

也就是：

- 先本地保留
- 暂不混入当前公开项目主线
- 如果后续要公开，先做抽象和清理

## 主要原因

### 1. 含有明确项目标识

当前 README 中出现了：

- 具体 GCP project id
- 具体 bucket
- 当前计费和运行说明

这些都更像个人当前环境配置，而不是通用公开模板。

### 2. 与当前公开项目主线不同

当前公开主线是：

- OpenClaw skills
- public-safe 协同规则
- GitHub / ClawHub / 飞书知识库同步体系

而 `scripts/vertex/*` 更像：

- 个人本地 AI 媒体生成工具链
- 独立实验脚本

### 3. 还未完成 public-safe 抽象

虽然代码本身不一定敏感，但文档和默认配置仍然带有个人环境痕迹。

## 如果后续想公开

建议先做这些动作：

1. 去掉具体 project id
2. 去掉具体 bucket
3. 改成环境变量占位写法
4. 改写 README，不再写当前个人计费和环境说明
5. 判断是否要拆成单独的公开工具目录或独立仓库

## 分级建议

### 当前阶段

- `C 级：仅本地保留`

### 完成抽象后

如果完成去标识和模板化改写，可重新评估为：

- `D 级：先抽象再判断`

再进一步通过后，才可能进入：

- `A 级：可公开发布`

## 一句话结论

`scripts/vertex/*` 目前更像个人环境下的本地工具脚本，不建议直接进入当前公开项目主线；若要公开，先做彻底模板化和去标识化。
