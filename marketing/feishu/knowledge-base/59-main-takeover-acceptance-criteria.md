# 59_main 接管验收标准

## 目标

判断 `main` 是否已经从“会回消息”进入“能稳定承接工作”的状态。

## 验收标准

### 1. 上下文稳定

`main` 应能够默认继承：

- 正式中枢群上下文
- 项目登记表
- 当前项目敏感度与公开边界

不应再默认：

- 扫描主目录
- 通过仓库名去猜本地路径
- 频繁要求重复提供已登记的项目信息

### 2. 回报稳定

`main` 的回报应稳定包含：

- `状态`
- `动作`
- `同步范围判断`
- `证据`
- `下一步`

多项目场景下，应按项目分段回报。

### 3. 脱敏稳定

`main` 不应在正式群里直接回显：

- 本机绝对路径
- `.openclaw` 目录
- 实例根目录树
- 原样 `git status`
- 未脱敏的内部结构明细

### 4. 判断稳定

`main` 应能够区分：

- 可公开进入 GitHub 的内容
- 仅飞书知识库沉淀的内容
- 仅本地保留的内容
- 需要你拍板的内容

不应默认：

- 看到改动就建议整批同步 GitHub
- 把“本轮不混提”直接等同于“永久忽略”

### 5. 路由稳定

`main` 应能够正确区分：

- 自己先做判断与回报
- 需要派给 `小果` 做公开同步
- 需要交给 `果爸` 做国内平台执行
- 需要你最终拍板

## 通过标准

如果连续数轮任务都满足下面 4 条，则可认为 `main` 已完成正式接管：

1. 不再跑偏到全局扫描
2. 不再泄露本地结构
3. 能做分批与分级判断
4. 能按规则把事情继续路由下去

## 未通过时的处理

如果仍出现以下情况，则视为尚未完全接管：

- 仍然反复要求已知项目路径
- 仍然回显本地结构
- 仍然把多类改动一把建议全发
- 仍然无法区分公开/知识库/本地

此时应：

- 继续在群里纠偏
- 记录问题类型
- 只修最小缺口，不重新推翻整套规则

## 一句话规则

`main` 不是“会回复”就算接管，而是要连续稳定地接住、判断、脱敏、分流，才算真正接管。
