# 62_GitHub协同群日审与上传规则

## 目标

让 `GitHub协同群` 里的代码相关动作不再靠临场判断，而是按固定节奏执行：

- 先审查
- 再分类
- 再决定是否提交
- 最后再决定是否要同步公开面

## 适用群

- `GitHub协同群`
- 当前 `chat_id` 已登记到 `special_projects`
- 群内默认入口：`main`

## 当前正确提交方式

当前这条链路是正确的：

`GitHub协同群 -> main -> 本地仓库修改/校验 -> commit / push -> 如需公开同步再派给 小果`

原因：

- `GitHub协同群` 当前挂在 `special_projects`，由 `main` 承接入口判断
- 代码修改、校验、git staging、commit、push 本来就应留在 `main`
- 只有 GitHub 展示、ClawHub、飞书知识库、对外文案同步才派给 `小果`

## 不正确的做法

下面这些不应作为默认做法：

- 让 `小果` 直接改仓库代码并提交
- 在未判断改动主题前直接 commit
- 把多个主题混成一次 push
- 把 handoff、临时记录、敏感本地文件顺手带进提交
- 因为看到当天有改动，就默认必须上传 GitHub

## 每日标准节奏

### 1. 先做日审

每天先回看：

- 今天围绕默认项目产生了哪些实际改动
- 这些改动是否属于同一主题
- 哪些应进 GitHub
- 哪些只进知识库
- 哪些只留本地

### 2. 再判断是否值得上传

只有满足下面任一条件，才建议进入 GitHub：

- 仓库首页、README、CHANGELOG、release 叙事需要更新
- 某个 skill 有实质提升
- 安装、示例、校验、结构化脚本更完整了
- issue / 文档 / 展示入口已经明显落后于当前仓库状态

如果只是：

- 临时 handoff
- 群内回执
- 未收口草稿
- 本地敏感配置

则不应直接上传。

### 3. 确认是否属于同一提交主题

提交前必须先判断：

- 是否是同一发布主题
- 是否会混入其他线程文件
- 是否需要拆成两笔以上提交

如果主题不一致，先拆，再提交。

### 4. 提交前最小校验

至少确认：

- 当前 `git status`
- 修改文件范围
- 是否已有未收口的他线程改动
- 是否需要 `./validate_repo.sh`
- 是否需要版本或同步范围说明

### 5. 提交后再判断公开同步

提交 / push 完成后，再判断是否还需要：

- GitHub 首页同步
- ClawHub
- 飞书知识库
- 对外文案

不要把“代码提交”和“公开同步”混成一步。

## GitHub协同群的默认判断规则

在 `GitHub协同群` 中，若未明确指定其他仓库，则默认按：

- 仓库：`guoma970/openclaw-oss-starter`
- 本地路径：`/Users/<your-user>/<private-path> project`
- 分支：`main`

若任务涉及代码或仓库文件：

- 默认由 `main` 承接

若任务只涉及公开同步：

- 由 `main` 判断后派给 `小果`

## 每日推荐指令

### 标准版

```text
请从 main 对 openclaw-oss-starter 做一次 GitHub 日审。
要求：
1. 先检查当前工作区和今天的实际改动。
2. 区分哪些应进 GitHub、哪些只进知识库、哪些只留本地。
3. 判断这些改动是否属于同一提交主题。
4. 如果值得上传，再按最小安全范围提交并 push。
5. 最后按“状态 / 动作 / 同步范围判断 / 证据 / 下一步”回报。
```

### 更短口令

```text
请从 main 做一次 openclaw-oss-starter 的 GitHub 日审，并按规则判断今天是否值得提交上传。
```

## 每日回报最低要求

回报里至少要有：

- 状态
- 今天识别到的改动范围
- 是否值得上传
- 是否已提交 / push
- 是否还需派给 `小果`
- 下一步

## 一句话规则

`GitHub协同群` 默认不是“看到改动就上传”，而是“先由 main 日审，再按主题安全提交，最后再判断是否要同步公开面”。
