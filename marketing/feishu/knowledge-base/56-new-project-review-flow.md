# 56_新项目出现时 main 的审核流程

## 目标

让 `main` 在发现新项目时，有一套稳定的审核流程，而不是直接决定是否公开发布。

## 适用范围

- 新出现的项目目录
- 新进入中枢群视野的项目
- 新登记到项目登记表中的项目

## 标准流程

### 1. 识别新项目

`main` 首先确认：

- 项目名称
- 本地路径
- 是否已登记到项目登记表

如果尚未登记：

- 先建议登记
- 不直接进入公开链路

### 2. 判断项目类型

`main` 先判断该项目更接近：

- 公开项目
- 内部推进项目
- 商业化敏感项目

参考：

- `49_项目敏感度与商业化规则`

### 3. 做第一轮分级建议

对于新项目，`main` 应先给出建议分级：

- `A` 可公开项目
- `B` 仅知识库沉淀
- `C` 仅本地保留
- `D` 先抽象再判断

### 4. 判断是否需要你拍板

以下情况默认需要你确认：

- 项目是否进入 GitHub
- 项目是否进入 ClawHub
- 项目是否属于商业化敏感
- 项目是否需要先抽象

### 5. 形成执行建议

`main` 只给建议，不直接越权拍板。

建议动作可以是：

- 进入 GitHub 主线
- 仅进入飞书知识库
- 暂时只保留本地
- 先抽象后再评估

## 一句话规则

新项目出现时，`main` 先审核、先分级、先给建议；是否公开，仍由你拍板。
