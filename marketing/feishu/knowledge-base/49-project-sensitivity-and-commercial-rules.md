# 49_项目敏感度与商业化规则

## 目标

明确不同项目不应共享同一套公开策略。

## 项目分层

### A 类：公开项目

特点：

- 可公开
- public-safe
- 适合 GitHub / ClawHub / 飞书公开知识库

当前代表：

- `openclaw-oss-starter`

### B 类：内部推进项目

特点：

- 正在开发
- 有沉淀价值
- 当前不默认公开

当前代表：

- `言町驾驶舱`

建议动作：

- 先走本地和内部知识库
- 是否公开由你单独拍板

### C 类：商业化敏感项目

特点：

- 与未来商业化强相关
- 不适合默认公开
- 要优先保护概念、路径、结构和竞争优势

当前代表：

- `羲果陪伴`

建议动作：

- 默认只做本地与内部沉淀
- 不自动进入 GitHub / ClawHub / 飞书公开知识库
- 若后续要公开，必须先做专门抽象和去商业化敏感处理

## main 的默认规则

### 对 A 类项目

- 可以默认判断是否同步 GitHub / ClawHub / 飞书知识库

### 对 B 类项目

- 可以做日常检查和内部沉淀建议
- 但不默认公开

### 对 C 类项目

- 可以做进展梳理和内部总结
- 但默认不进入公开发布链
- 重要节点要等你拍板

## 一句话规则

不是所有项目都应该被 `main` 推向公开链路；商业化敏感项目默认先保护，再判断是否抽象公开。
